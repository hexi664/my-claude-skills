# PRD 质量检查清单

> AI 在产出 PRD 后必须逐项自查，全部通过才能提交给老板确认。

---

## ✅ 必须通过（缺一不可）

### 结构完整性
- [ ] 1-10 章节全部填写，无空章节
- [ ] 概述中有明确的问题陈述（不是"用户可能想要"，是"用户正在遭遇"）
- [ ] 功能摘要一句话能让非技术人员理解
- [ ] 目标中有至少一个可量化的成功指标

### 用户故事
- [ ] 每个 US 是原子化的（一个 US = 一个独立行为）
- [ ] US 格式统一：作为[角色]，我想要[行为]，以便[价值]
- [ ] 每个 US 都有对应的验收标准（AC）
- [ ] 验收标准是可测试的（能直接写成 Playwright 断言）

### 页面设计
- [ ] 每个页面/弹窗都有 ASCII 线框图
- [ ] 线框图标注了导航栏/内容区/操作区/Tab Bar
- [ ] 有交互说明（点击/滑动/长按的响应）
- [ ] 有状态说明（空状态/加载中/错误状态）

### 数据与接口
- [ ] 列出了所有需要的接口
- [ ] 每个接口标注了 Demo 处理方式（写死/Mock/真实请求）
- [ ] 有 Mock 数据的 JSON 示例
- [ ] 数据模型字段说明完整

### 边界情况
- [ ] 覆盖了输入边界（空/超长/特殊字符/重复提交）
- [ ] 覆盖了网络异常（无网/超时/500/异常数据）
- [ ] 覆盖了设备差异（小屏/大屏/深色模式）
- [ ] 覆盖了权限（定位/相机/通知的拒绝处理）
- [ ] 覆盖了业务边界（未登录/免费用户/空数据）

### 范围控制
- [ ] Out of Scope 章节有内容（明确说了不做什么）
- [ ] 功能范围适合 Demo（不是完整产品规格）

---

## ⭐ 加分项（尽量满足）

- [ ] 场景表中标注了使用频率
- [ ] 有竞品中类似功能的参考（链接或截图）
- [ ] 考虑了 Widgetable 现有功能的一致性（交互模式/视觉风格）
- [ ] Mock 数据贴近真实数据结构
- [ ] 里程碑时间评估合理
- [ ] 开放问题列出了需要老板决策的点

---

## 🚨 红线（出现即打回）

- ❌ 验收标准写了"应该表现良好"之类模糊描述
- ❌ 没有 ASCII 线框图只有文字描述页面
- ❌ 接口章节只写了"需要后端接口"没说 Demo 怎么处理
- ❌ 边界情况章节为空或只写了"待补充"
- ❌ 功能范围膨胀到超出 Demo 合理规模
